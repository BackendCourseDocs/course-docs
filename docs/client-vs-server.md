# Client vs. Server (Frontend vs. Backend)

## مقدمه

در دنیای توسعه وب و برنامه‌های کاربردی، مفاهیم **Client** و **Server** پایه‌های اصلی معماری سیستم‌ها به شمار می‌روند. این دو مفهوم اغلب با **Frontend** و **Backend** هم‌پوشانی دارند، اما درک دقیق تفاوت‌های آن‌ها برای طراحی و توسعه سیستم‌های کارآمد ضروری است.

## تعاریف و تفاوت‌های Client و Server

<div dir="rtl">
<b>Client</b> (کلاینت) به دستگاه یا نرم‌افزاری گفته می‌شود که درخواست‌ها را به سیستم ارسال می‌کند و پاسخ‌ها را دریافت و نمایش می‌دهد. کلاینت معمولاً روی دستگاه کاربر (کامپیوتر، گوشی هوشمند، تبلت و غیره) اجرا می‌شود و نقش اصلی آن تعامل مستقیم با کاربر است. این بخش مانند مشتری در یک رستوران عمل می‌کند که سفارش می‌دهد و نتیجه را دریافت می‌نماید. کلاینت مسئول اجرای محلی کد، رندر کردن رابط کاربری، مدیریت ورودی‌های کاربر (کلیک، تایپ، لمس) و نمایش خروجی‌هاست. برای مثال مرورگر وب (Chrome، Firefox)، اپلیکیشن موبایل (مانند تلگرام یا اینستاگرام) و برنامه‌های دسکتاپ نمونه‌های رایج کلاینت هستند. پردازش‌های محلی در کلاینت باعث می‌شود انیمیشن‌ها، اعتبارسنجی ساده فرم‌ها و تغییرات صفحه بدون نیاز به ارتباط مداوم با سرور انجام شود که این امر سرعت و پاسخ‌گویی را افزایش می‌دهد.

کلاینت‌ها به دو نوع اصلی تقسیم می‌شوند: <b>Thin Client</b> (ساده و وابسته به سرور، مانند مرورگر وب) و <b>Thick Client</b> (پیچیده‌تر با پردازش محلی بیشتر، مانند برخی اپلیکیشن‌های دسکتاپ یا بازی‌ها). از نظر امنیتی، کلاینت نقطه ضعف محسوب می‌شود زیرا کد آن قابل مشاهده و دستکاری است (با ابزارهای توسعه مرورگر). به همین دلیل هیچ‌گاه نباید اطلاعات حساس (مانند کلیدهای API یا رمز عبور) را در سمت کلاینت ذخیره کرد. کلاینت همچنین باید با انواع دستگاه‌ها و اندازه‌های صفحه (Responsive) سازگار باشد و معمولاً از پروتکل HTTP/HTTPS برای ارتباط با سرور استفاده می‌کند.
</div>
<br>
<div dir="rtl">
<b>Server</b> (سرور) دستگاه یا نرم‌افزاری است که درخواست‌های کلاینت را دریافت، پردازش کرده و پاسخ مناسب را تولید و ارسال می‌کند. سرور مانند آشپزخانه رستوران است که سفارش را آماده می‌کند. این بخش معمولاً روی ماشین‌های قدرتمند در دیتاسنترها یا سرویس‌های ابری (مانند AWS، Google Cloud) اجرا می‌شود و به صورت ۲۴ ساعته فعال است. سرور مسئول کارهای سنگین و حساس مانند ذخیره‌سازی داده‌ها در پایگاه داده، اجرای منطق کسب‌وکار، احراز هویت کاربران، پردازش پرداخت‌ها، جستجوی پیشرفته و تولید پاسخ‌های پویا است. برای مثال وقتی شما در یک سایت جستجو می‌کنید، سرور الگوریتم را اجرا کرده، اطلاعات را از دیتابیس می‌خواند و نتیجه را آماده می‌کند. سرورها از وب‌سرورهایی مانند Nginx یا Apache، زبان‌های سمت سرور (Python، Java، Node.js، Go و غیره) و پایگاه‌های داده (MySQL، PostgreSQL، MongoDB) استفاده می‌کنند.

یکی از مهم‌ترین ویژگی‌های سرور <b>مقیاس‌پذیری</b> (Scalability) است؛ یعنی با افزایش تعداد کاربران می‌توان سرورهای بیشتری اضافه کرد یا از Load Balancer استفاده نمود. امنیت در سرور بسیار حیاتی است و از لایه‌های متعدد حفاظتی (فایروال، رمزنگاری، WAF، کنترل دسترسی) استفاده می‌شود زیرا تقریباً تمام داده‌های حساس در این لایه نگهداری می‌شوند. چالش‌های اصلی سرور شامل مدیریت تأخیر شبکه، هزینه‌های زیرساختی (برق، خنک‌کننده، پهنای باند) و مقاومت در برابر حملات (DDoS، SQL Injection و غیره) است.
</div>

## تعاریف و تفاوت‌های Frontend و Backend
<div dir="rtl">
<b>Frontend</b> (فرانت‌اند یا سمت کاربر) لایه‌ای از برنامه است که کاربر نهایی مستقیماً با آن سروکار دارد. این بخش مسئول خلق تمام آنچه کاربر «می‌بیند» و «با آن تعامل می‌کند» است: ظاهر بصری، چیدمان عناصر، انیمیشن‌ها، پاسخگویی به کلیک‌ها و لمس‌ها، فرم‌ها، منوهای کشویی، حالت تاریک/روشن، لودینگ‌ها، پیام‌های خطا و موفقیت و غیره. هدف اصلی فرانت‌اند ایجاد تجربه کاربری (UX) لذت‌بخش، سریع، زیبا و در دسترس برای همه کاربران (در موبایل، تبلت، دسکتاپ، حتی با صفحه‌خوان برای افراد نابینا) است. تقریباً تمام کد فرانت‌اند در مرورگر یا اپلیکیشن کاربر اجرا می‌شود و به همین دلیل به آن Client-Side نیز می‌گویند. سرعت بالای واکنش‌ها (بدون رفت‌وآمد مداوم به سرور) و امکان شخصی‌سازی لحظه‌ای (مثل تغییر رنگ تم بدون رفرش صفحه) از نقاط قوت بزرگ فرانت‌اند است. اما از طرفی چون کد در دستگاه کاربر در دسترس است، امنیت آن پایین‌تر بوده و نمی‌توان منطق حساس یا داده‌های خصوصی را اینجا نگه داشت.
</div>
<div dir="rtl">
<b>Backend</b> (بک‌اند یا سمت سرور) لایه‌ای است که کاربر آن را نمی‌بیند اما بدون آن تقریباً هیچ برنامه مدرنی کار نمی‌کند. این بخش مسئول تمام پردازش‌های پنهان، منطق اصلی کسب‌وکار، امنیت، ذخیره و بازیابی داده‌ها، محاسبات پیچیده، احراز هویت، مجوزدهی، ارسال ایمیل/نوتیفیکیشن، پردازش پرداخت، جستجوی پیشرفته، پیشنهادهای هوشمند (recommendation engine)، مدیریت کاربران همزمان (concurrency) و خیلی کارهای دیگر است. بک‌اند روی سرورهای قدرتمند (یا سرویس‌های ابری) اجرا می‌شود و به آن Server-Side می‌گویند. اینجا جایی است که داده‌های حساس (رمز عبور، اطلاعات بانکی، محتوای خصوصی) ذخیره و پردازش می‌شوند. بک‌اند باید بسیار امن، مقیاس‌پذیر و مقاوم باشد چون ممکن است همزمان هزاران یا میلیون‌ها درخواست را مدیریت کند.
</div>

### تفاوت‌های کلیدی Frontend و Backend 

| ویژگی                        | Frontend (فرانت‌اند)                                                                 | Backend (بک‌اند)                                                                    |
|-------------------------------|---------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------|
| مکان اجرا                    | مرورگر کاربر یا اپلیکیشن موبایل/دسکتاپ                                           | سرور (فیزیکی، مجازی یا ابری)                                                     |
| زبان‌ها و فناوری‌های اصلی   | HTML, CSS, JavaScript / TypeScript, React, Vue, Angular, Svelte, Next.js, Nuxt, Astro, Tailwind, Shadcn/ui, Framer Motion | Node.js, Python (Django/FastAPI), Java (Spring), Go, Ruby (Rails), PHP (Laravel), .NET, Rust (Actix) |
| تمرکز اصلی                   | UI/UX، زیبایی‌شناسی، تعامل کاربر، انیمیشن، دسترسی‌پذیری (Accessibility)، ریسپانسیو بودن | منطق کسب‌وکار، امنیت، عملکرد، مقیاس‌پذیری، مدیریت داده، APIها، پردازش سنگین     |
| دسترسی به داده‌ها            | فقط آنچه از API دریافت می‌کند (محدود و خواندنی/نوشتاری کنترل‌شده)                | دسترسی کامل به دیتابیس، فایل‌سیستم، سرویس‌های خارجی، کش، صف‌ها و غیره          |
| امنیت                        | پایین – کد قابل مشاهده و دستکاری است (DevTools) – فقط اعتبارسنجی ظاهری            | بسیار بالا – رمزنگاری، JWT/OAuth، Rate Limiting، WAF، SQL Injection Prevention و غیره |
| سرعت پاسخ به کاربر           | بسیار سریع (اجرای محلی)                                                            | وابسته به شبکه + پردازش سرور (معمولاً کندتر از فرانت‌اند)                       |
| بار اصلی پردازشی             | CPU/GPU/RAM دستگاه کاربر                                                            | CPU/RAM/دیسک/شبکه سرورها                                                          |
| تست و دیباگ                  | بیشتر بصری، مرورگر DevTools، Lighthouse، Storybook، Cypress (E2E)                   | Unit/Integration، Postman/Insomnia، Jest/Go Test، Load Testing (k6, Locust)، Monitoring (Prometheus, Grafana) |
| ابزارها و اکوسیستم رایج     | VS Code, Figma, FigJam, Vite, Turborepo, ESLint, Prettier, Vercel, Netlify, Cloudflare Pages | Docker, Kubernetes, GitHub Actions, AWS/GCP/Azure, Redis, Kafka/RabbitMQ, PostgreSQL, MongoDB, Elasticsearch |
| چالش‌های اصلی                | سازگاری مرورگرها، عملکرد روی دستگاه‌های ضعیف، مدیریت State پیچیده، SEO (در SPA)   | Concurrency & Scalability, Database Optimization, Security Vulnerabilities, Downtime Prevention |
| مثال واقعی                   | انیمیشن لایک در اینستاگرام، Dark Mode سوئیچ، Infinite Scroll، Drag & Drop در Trello | ذخیره پست جدید، چک کردن اسپم کامنت، محاسبه الگوریتم پیشنهاد، پردازش پرداخت Stripe |

<b>نحوه ارتباط Frontend و Backend</b>  
در برنامه‌های مدرن، فرانت‌اند و بک‌اند معمولاً کاملاً جدا از هم توسعه داده می‌شوند و فقط از طریق <b>API</b> با یکدیگر صحبت می‌کنند. رایج‌ترین انواع API عبارتند از:

<ul>
    <li>REST (هنوز بسیار پرکاربرد)</li>
    <li>GraphQL (دقیقاً آنچه نیاز است را درخواست می‌کند – over-fetching و under-fetching ندارد)</li>
    <li>tRPC / gRPC (در پروژه‌های عملکردمحور)</li>
    <li>Server Components / Server Actions (در React 18+ و Next.js 13+ – مرز بین فرانت و بک‌اند محوتر شده)</li>
</ul>
در معماری‌های جدید (مانند <b>Jamstack</b>، <b>Edge Functions</b>، <b>Full-Stack Frameworks</b> مثل Next.js / Remix / SvelteKit / Nuxt) بخشی از منطق بک‌اند مستقیماً در کنار فرانت‌اند نوشته می‌شود (Server-Side Rendering، API Routes، Server Actions) که باعث می‌شود مرز بین دو لایه تا حدی محوتر شود، اما همچنان مسئولیت‌های اصلی هر کدام حفظ می‌گردد.
</div>
